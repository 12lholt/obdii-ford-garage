uv vs pip — how the flow actually works

Grounded in your obdii-ford-garage project. read top to bottom

The big shift: with pip you manage a virtualenv, install into it, and remember to write down what you installed. With uv, the project file is the source of truth — you edit dependencies through commands, and uv keeps a hidden .venv and an exact uv.lock in sync automatically. You rarely "activate" anything; you just say uv run ….

The mental model

pip — manual & imperative

You are the package manager. You create the venv, activate it, install, and manually keep requirements.txt honest. Nothing records exact versions unless you do it yourself.

uv — declarative & automatic

You describe intent ("this project needs obd"). uv resolves it, writes the exact graph to uv.lock, and makes the .venv match — every time, on every machine.

Same task, both ways

pip: start a project

# create + activate a venv yourself
python -m venv .venv
source .venv/bin/activate
# install, then remember to record it
pip install obd
pip freeze > requirements.txt

uv: start a project

# adds to pyproject.toml, makes .venv,
# writes uv.lock — all in one step
uv add obd

No manual venv. No activate. No freeze. The .venv and uv.lock are created/updated for you.

pip: clone & run someone's project

python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
# versions may drift from the author's
python -m obd_garage.cli --help

uv: clone & run someone's project

uv sync
# exact versions from uv.lock — identical
# to what your collaborator has
uv run garage --help

This is your collaborator's whole onboarding: uv sync, then uv run.

Add an optional groupseparate requirements-*.txt filesuv add --optional azure X

What uv touches for you

pyproject.toml
YOU + uv edit · commit it
Your intent: "needs obd>=0.7.2", plus the azure / diagnostics extras.
uv.lock
uv writes · commit it
The exact resolved graph (every version + hash). Guarantees everyone gets the same install.
.venv/
uv manages · git-ignored
The actual installed packages. Disposable — uv sync rebuilds it from the lock.

Rule of thumb: commit pyproject.toml + uv.lock; never commit .venv/. The lock is what makes "works on my machine" stop being a problem.

Your project, end to end

  1. Add a need: uv add obd — it lands in pyproject.toml, the lock updates, the venv updates. One command, three things in sync.
  2. Group optional tools: uv add --optional diagnostics python-can cantools … — kept out of the core install so the at-the-car logger stays lean.
  3. Keep it current: uv lock --upgrade bumps to the newest versions your constraints allow (this is how python-can went 3.3.4 → 4.6.1).
  4. Run things: uv run garage snapshot -v bmw-535xi-2012 or uv run pytest — uv ensures the env matches the lock before running.
  5. Collaborator clones it: uv sync --extra diagnostics gives them a byte-for-byte identical environment. No venv setup, no version drift.
One sentence to remember: pip installs packages into a venv you babysit; uv keeps a project, its lockfile, and its venv in sync so you can just uv add, uv sync, and uv run.
